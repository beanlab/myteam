## Scenario Document Structure And Writing Standards

Summary:
The group reviewed revised scenario documents and discussed a standard
structure for them. The adopted structure is overview, context,
behavior, and related scenarios. Behavior can include subsections
such as examples, edge cases, and constraints. The overview should be
a short abstract that explains what the feature is and what it does
at a high level. Context should explain the world the feature lives
in and why the feature exists.

The group considered whether context should come before overview.
They settled on keeping the brief overview near the top, treating it
like an abstract, with context following it.

Behavior should focus on observable, user-facing effects: inputs,
outputs, and what happens when the feature is used. Examples and
edge cases should be allowed inside behavior because they are often
the best way to explain specific behavior. Required standalone
sections for examples or edge cases may encourage agents to invent
filler content.

Constraints were discussed as hard rules about allowed and disallowed
use, including valid configuration, required formats, and error
conditions. The group noted that constraints may also fit naturally
under behavior because they define the bounds of the feature.

Designs Discussed:
- Use overview, context, behavior, and related scenarios as the core
  scenario structure.
- Allow behavior subsections such as examples, edge cases, and
  constraints.
- Treat overview as a concise abstract rather than a replacement for
  context.
- Put examples and edge cases inside behavior when they clarify the
  feature.
- Keep related scenarios as their own section.

Decisions:
- Keep a brief overview near the top of scenario documents.
- Use context to explain the problem space and current limitations.
- Allow examples, edge cases, and possibly constraints to be part of
  behavior instead of forcing them into separate required sections.
- Put constraints under the behavior section rather than keeping them
  as a separate top-level section.
- Adopt this standard scenario template: overview, context, behavior,
  and related scenarios.
- Avoid scenario text that only makes sense to someone already
  embedded in the project.

## Human Judgment, Agent Code, And Context Quality

Summary:
The group discussed the difference between agent output that is
present and output that is good enough to trust. Dr. Bean criticized
vague
generated phrasing such as "missing updates from production changes."
A clearer explanation would describe a production configuration change
that adds a supported setting while the local configuration lacks that
setting, causing local runs to fail.

Dr. Bean argued that low-quality text is still context, but it is poor
context. It can distract future readers and agents, and accumulated
sloppy context may make future agent work worse. The same concern
applies to code. Generated code may pass tests while still being
verbose, error-prone, or structured in a way that allows later bugs.

The discussion compared current agent-development practices with
approaches where engineers merge pull requests based mostly on
evaluation results. Dr. Bean was skeptical that teams can stop caring
about code quality yet. Agents are trained on human code and operate
from observed context, so readable and hard-to-misuse code still
matters.

The example discussed was a context-manager-style `with` statement
that guarantees exit behavior. Code that centralizes required cleanup
can exclude whole classes of error, while code that relies on repeated
manual cleanup can lose that behavior during later edits.

Designs Discussed:
- Maintain a high writing standard for scenario documents because
  future agents will consume them as context.
- Prefer code structures that make important behavior unavoidable.
- Use evaluations, but do not treat passing evaluations as a complete
  substitute for engineering judgment.

Decisions:
- Keep caring about code and prose quality even when agents generate
  the first draft.
- Prefer explicit, concrete explanations over compressed phrases that
  require project-specific prior knowledge.
- Treat clean context as a contributor to reliable future agent work.

## Scenario Interviewing And Iterative Implementation Workflows

Summary:
The group discussed whether scenario documents should be written by
humans directly or produced through an agent-guided interview. Dr. Bean
suggested that an agent might be most useful as an interviewer: asking
what the human means, surfacing edge cases, prompting for missing
details, and formatting the document. In this model, the agent prompts
content rather than inventing it.

Christian described a Rubber Duck prompting lesson where repeated
model behavior was corrected by targeting it directly in the prompt.
The model used deterministic language that did not fit the intended
student dynamic, so Christian added a rule to avoid that phrasing.
This worked, though prompt-level fixes may need review after model
upgrades.

Tyler connected scenario writing to his work on an autograder library.
He wrote a scratch `demo.py` to express the user experience he wanted
from the library before the implementation existed. As he wrote the
demo and revised it, the desired interface and separation of concerns
became clearer. The point was that design often evolves through
writing, reading, and revising.

The group concluded that feature development should not assume a
strictly linear pipeline from scenario to tests to implementation. A
scenario can look good until implementation reveals that the design is
awkward or overly complex. In that case, the workflow should allow the
team to return to the scenario, revise it, and implement again.

Designs Discussed:
- A scenario interviewer agent that asks the human one question at a
  time.
- A reviewer agent that examines the conversation and scenario
  document for gaps.
- A document-maintainer agent that updates the scenario document from
  the interview.
- A workflow that can loop from implementation review back to scenario
  design or implementation revision.

Decisions:
- Treat scenario creation as an iterative design process, not a
  one-shot document-generation step.
- Prefer interviewer-style agents for extracting design intent from
  humans.
- Future workflows should support loops back to scenarios when
  implementation reveals design problems.

## Workflow Resume, Forking, And Session ID Discovery

Summary:
The group reviewed a pull request for workflow resume support.
Tyler explained that Codex can be asked to report its own thread
ID by injecting instructions into the objective prompt. Codex can
retrieve that ID from an environment variable, but Pi does not expose
the same simple mechanism.

They discussed whether the `myteam` harness could determine the
session ID without agent cooperation. The group inspected Codex
session files and found rollout metadata, session IDs, prompts,
messages,
permission policy, approved commands, skills, and other injected
context. They also noted that a large amount of prompt context can be
included even when irrelevant to the current repo.

Dr. Bean proposed injecting a unique nonce or workflow step ID into the
agent prompt. The harness could then scan recent session files for
that unique value and recover the associated session ID
programmatically. This avoids requiring the agent to report its own
session ID.

The group also discussed forking sessions. If two agents need to
reflect on the same conversation, the safe default is to fork the
session rather than having both agents resume and mutate the same
session. True resume may still be useful for efficiency, but it should
be used deliberately.

Designs Discussed:
- Inject a unique workflow step ID into the agent prompt.
- Scan recent session files for that ID to recover the session ID.
- Support session forking so multiple agents can reflect on the same
  prior conversation independently.
- Make forked sessions the safe default, with non-fork resume as an
  explicit option.

Decisions:
- Treat session forking as the safer default for multi-agent
  reflection.
- Consider nonce-based session discovery as an alternative to asking
  the agent to report its own session ID.
- Add session forking and programmatic session discovery as future
  workflow features.
