## Session Startup From Roles, Skills, And Workflows

Summary:
The group discussed making workflow startup more flexible and
less dependent on a single fixed prompt path. The goal was to let
`myteam start` launch a role, skill, or workflow directly, while
preserving the current session context when needed.

They also discussed a configurable default agent/model setting so
the startup path does not have to hard-code Codex. The current
`output` requirement for exploratory workflows should be optional,
since some workflows are meant to continue until the user stops
them.

Designs Discussed:
- `myteam start` as the primary session entry point.
- Role and skill startup from the same command family.
- A configurable default model in a local config file.
- Optional `output` for exploration-style workflows.

Decisions:
- Make workflow startup more flexible.
- Treat the default model as a user-configured setting.
- Allow exploratory workflows to run without requiring output.

## Quest-Driven Workflow Design And Delegation

Summary:
The discussion focused on simplifying workflow code by leaning on
Quest as a durable execution framework. The speaker described how
Quest replays completed steps so a workflow can resume after a
restart without re-running already-finished logic.

The group explored a delegated workflow model where a running
workflow can suspend, launch another workflow, and then resume with
the returned result. This was framed as a call-stack-like system
that would keep nested workflows from competing with each other.

They also discussed making workflows read more like instruction
manuals than manual state machines. The goal is to keep business
logic visible and avoid tracking too many moving parts in code.

Designs Discussed:
- Use Quest to replace manual state tracking where possible.
- Treat workflow steps as reusable invocations.
- Suspend a parent workflow while a child workflow runs.
- Resume the parent workflow with the child result.
- Model nested workflow execution as a stack.

Decisions:
- Push toward Quest-based execution for workflow code.
- Keep the workflow behavior explicit and step-oriented.
- Reduce manual state machine logic when possible.

## Debugging Practice Duck And Rubric Tuning

Summary:
The team reviewed the debugging practice duck and tested the
rubric prompts against sample code. The current version works, but
the prompts need to ask about the intent of the code more clearly.

The speaker noted that the evaluator should focus on what the code
was meant to do, not just what a line appears to do in isolation.
They also discussed how rubric state is tracked internally so the
system does not lose progress or repeatedly ask about the same
item.

The rubric content itself may need pruning or simplification so the
experience stays focused and easier to maintain.

Designs Discussed:
- Keep rubric state inside the workflow.
- Ask intent-focused debugging questions.
- Reduce rubric ambiguity by tying prompts to code context.
- Simplify the rubric where possible.

Decisions:
- Keep the duck running for now.
- Tighten rubric wording to focus on code intent.
- Avoid overly broad or confusing debugging prompts.
