## Session Startup From Roles And Skills

Summary:
The group discussed making `myteam start` the primary entry point
for sessions. The goal is to start a role or skill directly, without
forcing the user to type `Codex` or provide a separate objective
prompt every time.

They talked through the root `.myteam/role.md` file as the default
startup context when `myteam start` is run with no arguments. They
also discussed a general rule that a file path ending in `.py`
should run as a workflow while a folder containing a role or skill
should start that role or skill. The startup path should reuse the
existing role output from `myteam get role` and inject that content
into the session.

Designs Discussed:
- `myteam start` as the default session entry point.
- Starting the root `.myteam/role.md` when no specific role is given.
- Starting a role or skill directly from a folder path.
- Injecting the `myteam get role` output into startup context.

Decisions:
- `myteam start` with no args should use the root `.myteam/role.md`.
- `myteam start` should start the requested role or skill directly.
- Session startup should reuse role-loading logic instead of
  duplicating it.

## Workflow Modes For Explore, Plan, Implement, Conclude

Summary:
The discussion focused on separating work into smaller workflow
modes instead of relying on one large workflow. The proposed set was
`explore`, `plan`, `implement`, and `conclude`, with `commit` as a
possible follow-up workflow.

`Explore` is for uncertain problems where the human needs help
understanding options. `Plan` is for turning a chosen direction into
an implementation strategy. `Implement` is for straightforward work
where the desired outcome is already known. `Conclude` is for final
review, cleanup, and wrap-up once the work is essentially done.

The group also discussed a workflow that ends with an explicit
feedback prompt to the student or user, so each completed run ends
with a small programmatic question about usefulness.

Designs Discussed:
- Small workflows for distinct phases of work.
- An `explore` role or workflow for option discovery.
- A `plan` role or workflow for implementation strategy.
- An `implement` workflow for direct execution.
- A `conclude` workflow for final review and wrap-up.
- A closing feedback step at the end of a workflow.

Decisions:
- Prefer several small workflows over one large master workflow.
- Treat `conclude` as a real workflow, not just a review step.
- Add an end-of-workflow feedback prompt.

## Platform Direction For AI Classroom Tools

Summary:
The conversation broadened into platform design. The group argued
for an API-first platform with a consistent data model, clear role
and scope rules, and thin facades on top of a strong core. They were
critical of systems that let browser UI and hidden admin APIs drift
apart, since that creates inconsistent behavior and makes the system
hard to extend.

The group also talked about zero-trust service identity and a
generic compute layer that can support many experiences. The idea is
to make the core system simple and stable, then build specialized
features like grading, messaging, quizzes, and conversations on top
of it.

Designs Discussed:
- API-first platform design.
- One core model for data, roles, and scopes.
- Thin facades for specific use cases.
- Zero-trust service identity across the platform.
- A generic compute layer that specialized tools can call into.

Decisions:
- Favor a stable core platform over many one-off interfaces.
- Keep the public interface focused on the data that flows through
  the system.
- Allow specialized experiences to sit on top of a shared base.
