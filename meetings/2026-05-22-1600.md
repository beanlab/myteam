## `myteam` Role Startup Semantics

Summary:
The discussion focused on how `myteam start` should behave when loading a
role. The main concern was avoiding accidental execution when the user or
another agent only wants context. The group worked toward a model where the
loaded document clearly indicates whether it is meant to yield to the human
or proceed directly with work.

The conversation settled on the idea that the role text itself should control
whether the agent goes or yields, rather than inferring that behavior
implicitly from the caller. That keeps the behavior explicit and makes the
role file the source of truth for startup semantics.

Designs Discussed:
- Let the role document control whether startup yields or proceeds.
- Use a clear distinction between a context-only load and a task-starting load.
- Treat `myteam start` as a convenience wrapper around agent execution.
- Avoid implicit behavior that guesses whether an agent should act immediately.

Decisions:
- Make startup behavior explicit rather than inferred.
- Keep the role markdown as the primary place that determines go versus yield behavior.
- Prioritize finishing the role-startup behavior so it is predictable and scalable.

## Prompt, Context, And Task Structure

Summary:
The group spent a long time clarifying the distinction between context,
inputs, prompts, and tasks. The emerging model was that roles and workflows
should look more like functions: a stable instruction body, a small set of
runtime inputs, and a defined output shape.

They leaned toward treating role text as the prompt body and using front
matter for structured metadata. That would let the system inject runtime data
cleanly without making the document itself ambiguous about where task details
belong.

Designs Discussed:
- Separate static context from runtime input.
- Treat the markdown body as the prompt or instruction text.
- Use front matter for structured metadata such as name, description, input, and output.
- Make the output contract explicit when a document is expected to return data.

Decisions:
- Favor a function-like structure for roles and workflows.
- Keep the prompt/instructions in the text body.
- Put structured metadata in front matter when the document needs it.

## Workflow Input And Output

Summary:
The conversation converged on a cleaner `run agent` style interface: a
prompt, optional structured inputs, and an optional structured output schema.
The output schema can be appended automatically, which keeps the authored
document simple while still allowing machine-friendly return values.

The group also discussed having `load.py` contribute additional context that
gets injected into the prompt. The goal was to make the runtime behavior
deterministic and easy to reason about, rather than requiring agents to infer
where each piece of information belongs.

Designs Discussed:
- Pass a prompt and structured input separately.
- Inject `load.py` output into the prompt context.
- Append output schema instructions automatically when a document declares an output contract.
- Use the same pattern for both direct `run agent` calls and `myteam start`.

Decisions:
- Keep the call pattern close to a normal function signature.
- Append the output instructions automatically when the document declares an output.
- Treat the file contents as the authored prompt body.

## Immutable State And Rubric Refactoring

Summary:
In the final part of the conversation, the focus shifted to workflow
implementation style. The recommendation was to think in terms of immutable
state and function calls that return new state, rather than mutating tracking
data in place. That would make the rubric and workflow progression easier to
reason about.

The advice was to model progression as a series of explicit transformations.
The conversation suggested that this may simplify the refactor of the review
workflow and make the state machine more predictable.

Designs Discussed:
- Prefer immutable workflow state where possible.
- Return new state instead of mutating rubric tracking in place.
- Use explicit function calls to represent workflow progression.
- Keep mutable state limited and boxed in when needed.

Decisions:

## Target `role.md` Format

Summary:
The end goal for `role.md` is a document that behaves like a callable agent
definition. The transcript pointed toward a simple structure where the
markdown body is the prompt text, and the front matter carries the structured
metadata needed to run it.

In that model, `name` and `description` identify the role, optional `input`
describes runtime data the caller must supply, and optional `output`
describes the schema or return shape the agent should produce. The runtime can
then inject `load.py` output and any provided input into the prompt before
execution.

The same document should also encode whether the agent should yield or
proceed. That behavior stays in the text itself, so a role can explicitly act
like a context-only load, a subprocess that waits for input, or a task that
starts running immediately.

Designs Discussed:
- Put role metadata in YAML front matter.
- Keep the markdown body as the authored prompt.
- Add optional `input` and `output` fields to define the runtime contract.
- Append output-format instructions automatically when an output contract exists.
- Keep go-versus-yield behavior explicit in the role text.

Decisions:
- Aim for a `role.md` format that mirrors a function signature.
- Treat the prompt body as the main authored content.
- Use front matter for structured metadata and contracts.
