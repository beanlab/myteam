## Role Deprecation And The Skills / Tasks Shift

Summary:
The discussion started with the observation that the current defaults
had been simplified a lot and that a large amount of code and tests
had been removed. That led into a broader design conversation about
the future of roles. The conclusion was that roles are no longer the
best abstraction for the system the team wants to build.

The group converged on a workflow-centric model where skills and
tasks take over the useful parts of the old role behavior. Roles were
described as a middle ground that creates confusion and overhead,
especially when agents need to know what they are supposed to do
ahead of time. The preference was to make the system clearer by
removing role-specific behavior and moving that information into
skills, tasks, and startup documents.

Designs Discussed:
- Keep roles as a distinct concept with separate loading rules.
- Deprecate roles and move users toward skills and tasks.
- Treat roles as a confusing middle layer that adds token cost and
  complexity.
- Use skills for discoverable knowledge and tasks for explicit work.

Decisions:
- Deprecate role-based behavior.
- Move the project toward skills and tasks as the primary concepts.
- Remove role listing and role-specific startup assumptions over time.
- Remove role support in one major release after warning users.

## Direct Startup Of Markdown, Python, And YAML

Summary:
The main technical design conversation focused on how `myteam start`
should behave when pointed at different kinds of files. The preferred
model was to let the user start a markdown file, a Python file, or a
YAML file directly, with each file type serving as a clear execution
unit. The path should include the extension. That makes the system
more predictable than inferring behavior from names alone.

The team compared this to the old pattern of starting a directory and
then looking for a `load.py` file inside it. The new direction was to
make the file itself the primary unit of execution when possible.
That said, roles and skills should remain startup targets for now.
That lets a markdown file behave like a reusable task document with
structured front matter, while Python remains the place for custom
logic and data loading.

For markdown files, the front matter should stay permissive. After
`name` and `description`, the remaining fields should be optional, as
they are for roles and skills today. The allowed fields should match
the signatures of `run_agent` and `AgentContext`.

Designs Discussed:
- Start a markdown file directly as a task document.
- Start a Python file directly for custom execution logic.
- Start a YAML file directly for workflow definitions.
- Continue to support starting roles and skills for now.

Decisions:
- Prefer direct file startup over directory indirection.
- Treat markdown files as a convenient way to test or run a single
  step.
- Keep Python files available for richer prompt generation and data
  gathering.
- Preserve existing role and skill startup behavior for now.

## Discoverability Of Skills And Tasks

Summary:
The transcript spent a long time on how agents should discover what
they can do. The direction that emerged was to make skills a clear,
discoverable concept and to add similar support for tasks. The idea is
that an agent should be able to ask for a list of skills or tasks and
then load only the relevant pieces into the prompt.

The team also discussed making `myteam get skills` a built-in entry
point that explains how skills work. That would let an `agents.md`
style file or similar startup document preload skill guidance without
hard-coding every skill into the agent prompt. Tasks would follow a
similar pattern, with machine-readable front matter used to mark
something as runnable and to describe its input and output contract.

Workflow definitions in YAML, Python, and markdown should keep `input`
optional. If an `input` field is present and the caller does not
provide input, the command should error and allow a retry. The same
existing error-and-retry behavior used for role and skill input can be
reused here.

Designs Discussed:
- Add a built-in `myteam get skills` command.
- Add a built-in `myteam get tasks` command.
- Add `skills` and `tasks` arguments to `run_agent`.
- Use front matter to mark runnable markdown or Python content.
- Keep `input` optional, but enforce it when present.
- Let startup prompts preload only the skills or tasks that matter.
- Add `list_skills` and `list_tasks` helpers for workflow authors.

Decisions:
- Make skills discoverable through built-in help content.
- Add tasks as a parallel concept with explicit input and output.
- Use front matter to identify runnable task documents.
- Keep skills and tasks separate discovery paths.

## Release Planning And Migration Notes

Summary:
The later part of the meeting moved into release planning. The main
topic was how to stage the changes without confusing users. The
conversation pointed toward a major release that removes role-based
behavior, adds the new skill and task discovery commands, and updates
the initialization flow to match the new model.

There was also discussion about `myteam init` creating a `.config`
file and optionally an `agents.md` file. The purpose would be to make
the defaults editable while still supporting a workflow-oriented
project setup. The group also noted that a migration note needs to be
very explicit about what changes and what no longer works.

`agents.md` should remain optional. `myteam init` should offer an
interactive mode that asks whether the user wants workflows, desired
settings, or the older role-based setup that requires `AGENTS.md`.

Designs Discussed:
- Use a major release to remove role support cleanly.
- Update `myteam init` to create editable defaults.
- Optionally create an `agents.md` file for classic agent startup.
- Provide an interactive `myteam init` mode for setup choices.
- Document the migration path clearly for users.

Decisions:
- Treat the role removal as a release-level migration item.
- Keep the initialization flow simple and editable.
- Write migration notes that explain the new command set clearly.
