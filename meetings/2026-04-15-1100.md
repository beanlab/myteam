# Meeting Notes

Date: 2026-04-15 11:00

## Summary

The meeting focused on whether workflow orchestration should live
inside `myteam` or move into a separate, more generic project. The
group also discussed AI usage costs, the value of deterministic
workflow steps, a live workflow demo, and reflections on what the
class taught about software craftsmanship and AI-assisted
development.

## Main Points

- Workflows may be better as a separate project or plugin rather
  than a core `myteam` feature because they are an execution concern,
  not a content-management concern.
- A generic workflow engine with light `myteam` integration was
  compared to Jira and Confluence style integration between related
  tools.
- The discussion favored an `exec`-mode oriented workflow engine
  with a pluggable `talk to user` interface so different agent tools
  and UIs can share the same backend model.
- The group reviewed recent AI usage totals, including roughly
  922 million tokens and about $466 spent over two weeks across
  eight users.
- Frontier models were described as powerful but expensive, which
  reinforces the appeal of workflow systems that reset context and
  reduce repeated reasoning overhead.
- Deterministic workflows were seen as a way to improve reliability,
  control costs, and reduce agent decision overhead.
- A workflow demo showed YAML-defined steps, dependency tracking,
  persistent runs, streamed output, isolated workspaces, and
  interactive pauses.
- The demo feedback stressed reducing scrolling, collapsing old
  steps, and designing future demos around one simple idea that is
  easy to grasp quickly.
- The meeting closed with class reflections about writing code for
  future readers, collaborating through PRs, and using AI to amplify
  capable builders rather than replace judgment.

## Decisions

- Continue treating `myteam` primarily as a progressive-disclosure
  and content-distribution tool.
- Explore workflows as a separate project or plugin with only light
  `myteam` integration.
- Favor explicit, step-based workflow structure over agent-managed
  control flow when reliability and efficiency matter.

- Should workflows be a `myteam` plugin or a completely separate
  repository?
- What is the right abstraction boundary between workflow execution,
  agent tooling, and user-interface integrations?
- How much value can smaller or open models deliver once workflows
  encode more of the reasoning structure?
- Will the market increasingly reward developers who can build
  reliable harnesses and maintain code quality under AI acceleration?
