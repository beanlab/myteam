## Documentation-Only Merge Process And CI Skipping

Summary:
The meeting opened with a review of how to merge non-code changes
such as meeting minutes without triggering unnecessary CI work. The
group discussed using recognized skip markers in commit messages so
GitHub Actions does not rebuild unchanged code paths for
documentation-only pull requests.

Decisions:
- Use a recognized skip-CI marker in the last commit message for
  non-code merges when avoiding a rebuild is appropriate.
- Treat documentation-only pull requests as valid cases for skipping
  unnecessary CI execution.

Tasks:
- [Tyler] Use a skip-CI commit marker for documentation-only pull
  requests when appropriate.
- [Tyler] Confirm which skip markers the repository currently honors
  and use one of the supported forms consistently.

Open Items:
- Which skip-CI phrase should be treated as the project's standard?
- Should the contribution process document when skipping CI is
  acceptable?

## Workflow TTY Filtering And Session Architecture

Summary:
Most of the meeting focused on the workflow runner's terminal
integration and output filtering. Tyler described intermittent
failures where filtering removes too much output, prevents
completion objects from being recognized, or otherwise breaks the
workflow. The discussion concluded that the output-processing logic
is too entangled with the TTY session wrapper and should be split so
the low-level session code remains stable while higher-level output
parsing and filtering evolve independently.

Decisions:
- Refactor the workflow session code to keep TTY handling isolated
  from output parsing and workflow-control logic.
- Keep the generator-based interaction model and move toward a
  clearer context-managed session wrapper.
- Prefer simpler control detection, such as explicit delimiter
  sequences around workflow results, over fragile parsing of
  filtered terminal output.
- Keep more raw output visible during development so workflow
  behavior is easier to inspect and debug.

Tasks:
- [Tyler] Refactor the PTY session wrapper so TTY state management
  lives in a dedicated context-managed object.
- [Tyler] Separate workflow-control parsing from display filtering
  so completion detection is not affected by presentation logic.
- [Tyler] Explore using explicit start and end delimiter sequences
  around workflow result payloads.
- [Tyler] Check where user input is echoed and determine whether it
  can be suppressed or should remain visible.
- [Tyler] Simplify the listener or filter interface so the chunk
  handling path is easier to reason about and maintain.

Open Items:
- Should workflow result blocks always remain visible in development
  mode, even if hidden later in a quieter default mode?
- Is a single responder interface enough, or will multiple
  listeners be needed for display, control handling, and logging?
- Should result parsing operate on raw bytes, normalized text, or a
  line-buffered intermediate form?

## Workflow Direction And Evaluation

Summary:
The conversation ended with a broader design reflection on the role
of workflows, harnesses, and code quality. The group discussed
whether workflow design itself may become the central programming
activity, both for product development and for teaching, and noted
that workflows may be easier to evaluate than general-purpose
harnesses because their outputs can be compared more directly
against expected results.

Decisions:
- Continue treating workflow design as a central area of product
  exploration.
- Use code clarity and reduced complexity as a key quality bar for
  workflow infrastructure.

Tasks:
- [Unassigned] Continue evaluating whether workflows should be the
  main frame for future teaching and demonstrations.
- [Unassigned] Explore how workflow outputs could be evaluated
  against expected results more directly than harness behavior.

Open Items:

  workflow design, or a combination of both?
- What is the best way to evaluate workflow quality and correctness
  in a repeatable way?
