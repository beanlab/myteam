# Concepts

- Step execution should be a reusable unit with workflow-specific
  input resolution kept at the workflow layer.
- Context managers are the right boundary for session and result
  channel lifecycle handling.
- Smaller built-in prompts preserve context budget and can improve
  behavior, especially for smaller models.
- Code is part of the specification because it captures decisions
  that are difficult to express fully in prose.
- Complex systems should grow through iterative complexity:
  start at pseudocode-level simplicity and add structure only when
  needed.
- Backlog tools remain useful for agents because they provide
  durable state, explicit actions, and clear workflow transitions.

## Workflow Runtime Refactor And Python Workflow Direction

Summary:
The main code review focused on Tyler's `myteam` workflow runtime
refactor. The current PR improved the terminal-session structure
and result tool integration, but the group agreed the boundary
between workflow logic and step execution still needs to move.
Input resolution, prior-step handling, and YAML interpretation
belong at the workflow layer, while `execute_step` should become a
reusable entry point for launching a single agent task with
prepared input and output requirements. This would make
Python-defined workflows easier to support and would also help with
partial or resumed execution.

Decisions:
- Keep the current PR as an incremental improvement.
- Move input-resolution logic out of step execution and into the
  workflow engine.
- Treat Python workflows as the near-term priority.
- Allow YAML workflow support to degrade temporarily if needed
  while Python workflow support is being established.

  layer and simplify `execute_step` to a reusable runtime unit.
- [Tyler] Keep the current workflow PR moving forward as an
  incremental cleanup rather than waiting for the full redesign.

Open Items:

  output schema, and agent selection?

  remain inside step execution versus move upward?

  supported public interface or become legacy-only?

## Role Loading, Startup Overhead, And Backend Adapters

Summary:
The team reviewed the cost of each step reloading role context
through `AGENTS.md` and `myteam get role`. That behavior is useful
for compatibility, but it adds latency, token cost, and repeated
reasoning on every step. The group discussed moving toward a model
where workflow execution can inject role context directly while
remaining compatible with existing `AGENTS.md`-based projects. The
team also reviewed backend adapter structure, confirmed the new
context-manager pattern looked correct, and agreed that a `pi`
backend can be added with minimal additional structure for now.

Decisions:
- Keep compatibility with `AGENTS.md` and role-based loading.
- Explore a minimal default workflow role or injected role context
  to reduce repeated startup overhead.
- Leave backend-adapter abstractions simple for now.
- Add a `pi` backend adapter without over-designing filtering or
  protocol layers yet.

Tasks:
- [Tyler] Explore a default workflow role or prompt-injection path
  that avoids repeated repo and role discovery on every step.
- [Tyler] Add a backend adapter for `pi`.
- [Tyler] Investigate why step input sometimes fails to paste on
  the first attempt.

Open Items:
- Should workflow execution inject role names, full role context,
  or both?
- Is there a clean marker that can tell `AGENTS.md`-driven sessions
  not to reload additional role context during workflows?
- Which output-filtering behavior should live in backend adapters
  once multiple tools are supported?

## Tutoring-Agent Redesign And Conversation Orchestration

Summary:
Christian reviewed work on the CS110 tutor or teacher bot, which
evaluates code against a rubric and tries to guide a learner
through discussion. The current role-playing approach sometimes
leads the student too aggressively or jumps ahead across rubric
items. The discussion shifted toward a design where no single agent
role-plays the student directly. Instead, multiple agents would
analyze the conversation, determine what the learner still does not
understand, propose the next question, and optionally rewrite that
question into learner-facing language.

Decisions:
- Move away from direct learner role-play as the primary design.
- Reframe the system around analysis of a conversation between
  teacher and learner.
- Let reviewing agents determine missing understanding and suggest
  the next question or conversational move.

Tasks:
- [Christian] Prototype a redesign where agents analyze the
  conversation and generate the next learner question rather than
  role-playing end to end.
- [Christian] Continue refining prompts so each step stays focused
  on one concept at a time.

Open Items:
- How should the system represent what the learner still does not
  understand at each stage?
- Should one final agent rephrase reviewer output into a natural
  learner question before it is sent?
- Does the analysis-first approach outperform direct role-play in
  realism and pedagogical quality?

## Iterative Development Process And Workflow Stages

Summary:
The meeting broadened into process design for agent-assisted
development. The current feature pipeline is better suited to
incremental backlog work than to heavy prototype-stage design.
Tyler argued for a tighter loop between planning and coding, where
small prototypes are created, reviewed, and refined instead of
trying to specify everything up front. The group discussed the
need for separate planning, implementation, and conclusion
workflows, along with better handoffs between sessions so work can
resume cleanly after context resets or interruptions.

Decisions:
- Favor iterative complexity over up-front complexity.
- Treat prototype-stage work as a different process from routine
  backlog implementation.
- Explore separate workflows for planning, implementation, and
  conclusion instead of assuming one uninterrupted session.

  progress across sessions.
