# Concepts

- Debugging exercises are a narrower and more deployable first
  target than open-ended conceptual tutoring.
- Tutor prompts should ask students to explain a concrete error in
  context, not merely apply a mechanical fix.
- Feature requests should not be treated as authoritative design
  specs; they should be refined through scenario and design work.
- Workflow state should live in durable backlog artifacts so work
  can resume across agents, sessions, and repositories.
- Development stages can be modeled as a state machine where the
  next step is explicit output from each role.

## CS110 Tutor Bot Direction And Debugging Exercises

Summary:
Christian reported on the CS110 tutor bot prototype. The current
conversation flow improved after changing the workflow so the
user-facing agent was less aware of the full process, but the
system still behaves too much like stock GPT and tends to ask
leading questions. The group discussed replacing direct role-play
with a third-party analysis approach: agents would inspect the
teacher-student conversation, identify what the student has not yet
understood, and generate the next student question.

The discussion then shifted to a narrower first feature: debugging
exercises. Instead of trying to probe broad conceptual
understanding immediately, the bot can show students broken code
and an error message, then ask them to explain what is wrong. This
fits common CS110 lab problems and gives students practice reading
stack traces, locating the failing line, interpreting the error, and
explaining the underlying programming rule. Logic errors and
state-based debugging were treated as valuable but out of scope for
this first version because they often require running code and
adding instrumentation.

Decisions:
- Focus the near-term tutor bot work on concrete debugging
  exercises.
- Start with failing code that produces identifiable runtime errors
  or compiler errors.
- Use precomputed error output from runnable snippets rather than
  asking the agent to reproduce the error live every time.
- Leave broader conceptual probing and oral-exam-style interaction
  for a later feature.
- Keep logic errors, such as wrong output without an exception, out
  of the first debugging exercise feature.

Tasks:

  debugging exercises where students explain concrete errors.
- [Christian] Build or curate snippets that fail in ways students
  commonly encounter.
- [Christian] Include the relevant error output with each exercise,
  with local paths cleaned so usernames or machine details are not
  exposed.
- [Christian] Check student responses for whether they identify the
  right line, interpret the error, and explain the programming rule
  involved.

  plausible student questions rather than agent summaries?

## Workflow Step API And Python Workflow Demos

Summary:
Tyler walked through changes to `myteam` workflow execution. The
main change was isolating `execute_step` so it takes a step
definition and returns a result, with demo workflows showing how
step results can feed later steps. A poem workflow works well, while
the feature pipeline still needs updated skills and prompts before
it uses the new structure effectively. The group reviewed the
interface and decided that `execute_step` should take explicit
keyword arguments rather than requiring callers to build a
dictionary, since dictionaries can still be unpacked by callers that
already have structured data.

The merge review covered versioning, changelog updates, and demo
workflow quality. The group rejected a jump to version 3 for the
experimental workflow API and kept the change in the 0.2 series.
The demo workflow still had an adapter that preserved old structure
instead of using the new `execute_step` interface directly, so that
was identified as cleanup work. The group also noted that Python
workflow files are currently loaded as modules, but should be
reviewed against the existing subprocess pattern used by role and
skill loading.

Decisions:
- Change `execute_step` to accept explicit keyword arguments rather
  than a single step-definition dictionary.
- Keep the workflow API change in the 0.2 version series because it
  affects an experimental feature.
- Merge the workflow step API work after the immediate review
  fixes.
- Review whether Python workflows should run in subprocesses to
  match existing `myteam get role` and load-file behavior.
- Consider adding a result helper such as `raise_on_failure` to make
  workflow failure handling cleaner.

## GitHub Projects Backlog Process

Summary:
Tyler reviewed Symphony and GitHub Projects as possible backlog and
automation systems. Symphony can automatically pick up issues,
drive agent implementation, and submit pull requests, but the group
decided that this is closer to a black-box or higher-autonomy
factory model than the current goal. The near-term target is a more
interactive development workflow where humans and agents iterate on
scenarios, design, implementation, and review.

The group favored GitHub Projects for the current backlog process.
Projects can connect multiple repositories, provide visibility into
who owns work and where it is in the broad process, and avoid
introducing a separate Linear account. The project board should stay
simple, with states such as backlog, in progress, and done, while
the detailed workflow state lives inside the ticket and workflow
artifacts. Tyler created or configured a GitHub Project named
`backlog` for BeanLab work and discussed adding Rubber Duck,
Roster, Quest, and related repositories.

Decisions:
- Use GitHub Projects as the near-term backlog system instead of
  Symphony or Linear.
- Keep project-board status coarse-grained, focused on ownership and
  whether work is available, active, or done.
- Store detailed scenario, design, implementation, and review
  context in the ticket rather than in many project-board columns.
- Build skills or instructions that teach agents how to read and
  update backlog tickets consistently.

Tasks:

  repositories.
- [Tyler] Add Rubber Duck, Roster, Quest, and related repositories
  to the backlog project where permissions allow.
- [Tyler] Create a skill or shared instruction set for connecting to
  GitHub Projects and updating tickets.
- [Tyler] Define the standard structure of a backlog ticket so
  agents can read and update sections without corrupting context.

## Development Workflow Stages And Durable Ticket Context

Summary:
The group discussed how agent-assisted development should move
between scenarios, design, implementation, review, and wrap-up. A
feature request should begin as a statement of desired behavior, but
it may need scenario work before design. Design should happen before
code changes and should remain conceptual. Implementation is where
files are edited and tests are run. Review may send work back to
implementation, design, or even scenarios if the team discovers a
missing case or a wrong approach.

The proposed workflow resembles a state machine. Each stage would
produce output that includes the next role or stage to run. The
natural next step might be implementation after design, but the user
could redirect the workflow back to design or scenarios. The
important durable state should live in the backlog ticket: scenarios
identified, design decisions made, files changed, interfaces added,
review feedback, and remaining concerns. This allows another agent
or session to resume work without relying on hidden conversation
context.

Decisions:
- Treat scenarios, design, implementation, and review as workflow
  stages that can loop back as needed.
- Keep code-writing inside the implementation stage; design remains
  conceptual.
- Store accumulated feature context in the backlog ticket so it can
  survive session boundaries.
- Let workflow steps emit the next stage explicitly, while allowing
  the user to redirect the flow.
- Reconsider application documentation so it captures high-level
  contracts and scenarios rather than duplicating code behavior.

Tasks:
- [Tyler] Prototype a development workflow that can move between
  scenarios, design, implementation, and review.
- [Tyler] Define what each stage should add back to the backlog
  ticket.
- [Tyler] Explore scenario documents as durable contracts that tests
  can be built against.
- [Tyler] Revisit application documentation strategy to reduce drift
  between prose and code.

Open Items:

- How should the workflow prevent an agent from skipping ahead to

- What review checks should compare repository state against the
