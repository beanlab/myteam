## Rubber Duck Debugging Conversation And Rubric Generation

Summary:
Christian demoed the current Rubber Duck debugging conversation
prototype. The system has been rewired so it is no longer a
multi-agent process; a single user-facing agent now drives the
conversation using the conversation-review model. The current version
is working better and is not leading with overly direct questions in
the same way as before.

The demo focused on debugging help. The student or TA can ask what
error the program is seeing, and the system provides a sample error.
Tyler asked that the displayed sample error be formatted in backticks
so it is visually easier to read. The conversation then asks a
question in all cases. The rubric logic is governed by two required
parts: the learner must identify why the code is not working and the
associated fix. If the answer includes only one of those parts, the
system should ask for the missing part.

The group discussed why both pieces matter pedagogically. Students
can often discover a mechanical fix, such as wrapping a string in
`int`, without understanding why it works. The debugging conversation
should require both the programming principle and the concrete change.

Designs Discussed:
- Keep the current single-agent conversation approach for the demo.
- Format traceback or sample error text as code so it is visually
  distinct.
- Treat each rubric item as requiring both an explanation of the
  principle and a fix.
- Ask follow-up questions when the learner supplies only the reason
  or only the fix.

Decisions:
- Keep iterating on the single-agent conversation design.
- Require both the underlying principle and the concrete fix in
  debugging answers.
- Format sample errors and tracebacks as code blocks or inline code
  where appropriate.

## `myteam` Workflow API Naming And Rubricizing Scripts

Summary:
Christian described a planned script that would let TAs provide a
bad Python program and automatically generate a debugging rubric from
it. The script would run the program, capture the latest traceback,
ask an agent to fix only that error, record the error, concept, and
fix in the rubric, then run the program again. The loop would
continue until the program is correct. The resulting rubric, along
with the original program text and the sequence of errors and fixes,
could then be checked into the Rubber Duck repo as configuration.

The initial implementation idea involved Docker so the agent could
run the script safely, especially on Christian's Windows machine.
Tyler proposed a simpler near-term path: put a script in the repo's
`scripts/` folder and run it locally. Since Rubber Duck currently
uses code-checked configuration rather than a persistent external
knowledge base, the generated rubric should become a repo artifact.

The group discussed using the `myteam` workflow runtime for the
agent loop. A workflow step could ask an agent to fix the one bug
demonstrated by the current stack trace and stop once the error
changes. The script would be responsible for running the program,
adding each unique traceback to the rubric, calling the agent for a
focused fix, and iterating until no traceback remains.

The API name `execute_step` was reconsidered. The group preferred
`run_agent`, because the operation is not just executing a structured
step; it is letting an agent run for one or more turns to accomplish
a task and then return.

Designs Discussed:
- A local rubric-generation script in `scripts/` that runs a Python
  file and accumulates traceback-based rubric items.
- A `myteam` workflow or Python API call that asks an agent to fix
  exactly one stack-trace error at a time.
- A loop that records the code, current error, concept, and fix, then
  reruns the program until it is correct.
- Renaming `execute_step` to `run_agent` to better describe the
  public API.

Decisions:
- Prefer a local script in the repo over a Docker-dependent first
  version.
- Use the `myteam` workflow runtime as a test case for this agentic
  loop.
- Rename or move toward renaming `execute_step` to `run_agent`.
- Keep Rubber Duck's deployable knowledge in checked-in
  configuration for now.

## Scenario Specification Skill And Scenario-Driven Tests

Summary:
The group reviewed early work on a skill for defining scenarios.
Christian had looked at existing approaches and found limited useful
prior art beyond LLM-as-judge style scenario evaluation. The group
compared the existing state/action/outcome framing with a preferred
context/action/outcome or precondition/action/guarantee structure.
The scenario should describe the context or relevant system state, an
action, and the expected guarantee after the action.

The proposed scenario outline included purpose, precondition, action,
guarantee, non-goals, and related scenarios. The group decided that
purpose and non-goals should be optional. A good file name may already
serve the purpose role, but an optional summary can help when the
scenario is broad. Non-goals or out-of-scope notes are useful when
needed but should not be required everywhere.

The group discussed how scenarios should work for CLI and library
interfaces. For CLI behavior, a scenario should be able to specify a
shell command and the expected outcome, including file-tree or system
state changes. For library behavior, a scenario should include code
snippets, such as importing `run_agent`, calling it with specific
inputs, and describing the expected return shape. Scenarios should not
overspecify storage details that are already encoded in code; for
example, whether state is stored in Markdown or SQL should remain an
implementation decision unless it is externally visible.

The scenario skill currently lives in a roster area and is not ready
to adopt broadly. The next step is to play with it, test whether it is
useful, and then move it into rosters and download it into `myteam`
when it is good enough. Once scenarios are available, the team expects
to refactor tests toward scenario-driven end-to-end coverage,
including at least one test that runs a real workflow with an
installed cheap agent model.

Designs Discussed:
- Scenario structure based on context, action, and outcome.
- Equivalent framing using precondition, action, and guarantee.
- Optional fields for purpose, non-goals, and related scenarios.
- CLI scenarios that include shell commands and expected state
  changes.
- Library scenarios that include code snippets and expected return
  shapes.
- Tests that exercise real workflows end to end through an installed
  agent.

Decisions:
- Use context/action/outcome or precondition/action/guarantee as the
  core scenario shape.
- Use `context`, `action`, and `outcome` as the final scenario section
  names.
- Make purpose and non-goals optional.
- Keep any scenario purpose text very short, about 10 words or less.
- Avoid duplicating implementation-storage details in scenarios unless
  they are part of the externally visible behavior.
- Experiment with the scenario skill before adopting it broadly.
- Move toward end-to-end tests driven by scenarios once the scenario
  format is stable enough.
