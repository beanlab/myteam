# CS 401R Class Discussion

Date: 2026-04-01 11:05

## Summary

The discussion covered three main areas:
meeting-audio transcription workflows,
privacy and ethics around recording and sending
conversations to AI systems, and a design
direction for workflow-driven role
orchestration in Codex and `myteam`.

The audio-transcription portion focused on
recording meetings, transcribing them into
Markdown, and using those records to generate
backlog items and project updates. The group
discussed always-on recording, chunking audio,
voice-activity detection, file sizes, and the
tradeoff between local transcription models
and paid hosted transcription.

Later, the discussion returned to engineering
work. The main technical focus was the Codex
`AppServer` architecture and the possibility
of building a workflow runner on top of it.
The idea is to have a separate entry point,
possibly inside `myteam`, that loads a
workflow definition from YAML, starts
`AppServer`, and advances through role-based
workflow steps while preserving a continuous
user experience.

## Main Points

- Meeting recordings can be useful if they are
  transcribed into repo-local Markdown and
  then used as durable context for future
  work.
- Hosted transcription is cheap enough to be
  practical for routine meeting capture, but
  local models remain attractive for
  privacy-sensitive use cases.
- Privacy risks increase when recordings
  include other people and are sent to
  external systems, even when enterprise
  agreements exist.
- A workflow runner could reuse existing
  Codex infrastructure instead of introducing
  a large parallel system.
- The proposed workflow model should describe
  steps by role, inputs, outputs, and
  references between step outputs.
- High-quality agent engineering depends less
  on role labels like "frontend" or
  "backend" and more on preserving the right
  information and review structure at each
  stage.

## Decisions

- Meeting transcripts should be treated as
  useful project artifacts that can later
  feed backlog creation and implementation
  planning.
- The proposed workflow system should prefer
  minimal new infrastructure and build around
  existing Codex `AppServer` behavior.
- Workflow outputs should be structured
  objects that later steps can reference,
  rather than only freeform conversation
  text.
- Development flow in `myteam` should
  continue emphasizing staged information
  quality: interface clarification,
  framework refactor, implementation design,
  test updates, implementation, and review.

- Define the YAML workflow format, including

- Investigate how `AppServer` manages threads
  and whether thread lifecycle can be
  controlled well enough to support workflow
  steps.
- Determine how a workflow step should be
  considered complete without relying on
  fragile semantic guesses.
- Review the open PR that further refines the
  `myteam` development process.
- Continue improving the `myteam` process
  around framework-first changes, interface
  accuracy, and review at each stage.

## Open Questions

- How should workflow completion be detected
  for a conversational thread?
- Can a thread be forced to emit a structured
  final output reliably enough for workflow
  orchestration?
- What is the right balance between local
  transcription for privacy and hosted
  transcription for convenience?
- What meeting-recording norms are acceptable
  when other people are being recorded?
