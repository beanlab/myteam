## Workflow Schema And Execution Model

Summary:
The group reviewed a workflow implementation that can use both
`exec` and app server, with structured JSON output in both paths.
They focused on schema naming, explicit dependencies, session reuse,
and whether `myteam` should standardize on one thread-management
path rather than maintain separate interaction models.

Decisions:
- Use `interactive` rather than `automatic` for workflow step
  behavior.
- Rename `additional context` to `input`.
- Favor explicit dependencies through referenced outputs rather than
  implied ordering or file-system side effects.
- Keep the possibility of keyed step definitions with dependency
  ordering open for future work.

- [Unassigned] Explore explicit dependencies, session reuse, and

## Role And Skill Creation Workflow

Summary:
The meeting also covered the friction in creating new roles and
skills. The discussion emphasized keeping shared creator helpers in
roster, relying on `load.py` and the file system for composition,
and making new-role and new-skill commands print created file paths
for fast editing.

Decisions:
- Keep the built-in surface small while creator tools are still
  evolving.
- Treat roles and skills as standalone documents that start at
  heading level one.
- Keep creator-oriented helpers in roster for now.
