## 0.2.27 migration

Version `0.2.27` turns workflows into tasks and updates the default `.myteam`
loaders/templates to surface that new structure.

### What changed

- `myteam start` now treats markdown-defined task files as the default runnable unit.
- The task system has dedicated child commands, including `myteam task start` and
  `myteam task result`, for nested task execution and completion reporting.
- The default root role loader now lists tasks alongside skills and roles so the active task tree
  is visible during startup.
- New task scaffolding and task definition templates were added, including YAML frontmatter for
  task settings.
- Existing references that still use older workflow terminology may need to be updated to task
  terminology in local `.myteam` copies, prompts, or wrapper scripts.

### How to migrate an existing `.myteam` folder

1. Refresh any copied root or role `load.py` files from the current templates so they import and
   call the task helpers used by this release:
   - `explain_tasks()`
   - `list_tasks(...)`
2. If you have local startup wrappers, confirm they still print the project tree and discover tasks
   from the project root in the same way as the current packaged templates.
3. Convert any runnable markdown workflow files into task files if they are meant to be started with
   `myteam start <path>`:
   - keep the task prompt in the file body
   - move task settings into YAML frontmatter
   - add or refresh any `agent`, `model`, `output`, `input`, or related settings as needed
4. Update any local prompts, docs, or role text that still refer to older workflow commands so they
   use the current task command names.
5. If your `.myteam` tree already matches the current templates, no structural file migration is
   required. In that case, only update `.myteam/.myteam-version` to `0.2.27`.
