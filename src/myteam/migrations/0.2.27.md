## 0.2.27 migration

Version `0.2.27` makes tasks the primary runnable unit and updates the default
`.myteam` templates to expose that structure at startup.

### What changed

- `myteam start` now treats markdown task files as the default thing to run.
- Task execution has dedicated child commands, including `myteam task start`
  and `myteam task result`, for nested runs and completion reporting.
- The default root loader now shows tasks alongside skills and roles during
  startup, so the active tree is visible immediately.
- New task scaffolding and task definition templates were added, including YAML
  frontmatter for task settings.
- Older workflow terminology may still appear in copied `.myteam` trees, prompt
  text, or wrapper scripts and may need to be updated.

### How to migrate an existing `.myteam` folder

1. Refresh copied root and role `load.py` files from the current templates.
   - The root loader should call `explain_tasks()` and `list_tasks(myteam, myteam, [])` alongside
     the existing skills and roles output.
   - Role loaders that were reduced to a minimal wrapper may need to be expanded back to the full
     current role template.
2. If you have local startup wrappers, confirm they still print the project tree from the project
   root and discover tasks the same way as the packaged templates.
3. Convert runnable markdown workflow files into task files only when they are intended to start
   with `myteam start <path>`.
   - keep the task prompt in the file body
   - move task settings into YAML frontmatter
   - add or refresh any `agent`, `model`, `output`, `input`, or related settings as needed
4. Update local prompts, docs, and role text that still use older workflow terminology or command
   names.
5. If your `.myteam` tree already matches the current templates, no structural migration is needed.
   Update `.myteam/.myteam-version` to `0.2.27` after confirming the tree is in sync.
