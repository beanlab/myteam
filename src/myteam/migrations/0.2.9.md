## 0.2.9 migration

Version `0.2.9` treats workflows as named assets inside `.myteam/workflows/`,
adds `myteam new workflow <name>` to scaffold them, and makes `myteam workflows
start <name>` resolve workflow definitions from that location.

### What changed

- A `workflows` directory is now expected under `.myteam`. Each workflow lives in
  `.myteam/workflows/<name>.yaml`, where `<name>` is slash-delimited (just like
  roles and skills) and does not include the `.yaml` extension.
- `myteam new workflow <name>` copies the built-in workflow template from
  `src/myteam/templates/workflow_template.yaml` into `.myteam/workflows/<name>.yaml`.
- `myteam workflows start <name>` validates the named workflow file and starts the
  runner using that YAML.
- The CLI still persists run state to `.myteam/workflow_runs/` and reuses the same
  `inputs`/`outputs`/`{from: ...}` validation.

### How to migrate an existing `.myteam` folder

1. Create a `.myteam/workflows` directory if it does not already exist:

   ```bash
   mkdir -p .myteam/workflows
   ```

2. Choose a slash-delimited workflow name such as `planning/launch` and copy the
   new workflow template into the matching file:

   ```bash
   cp src/myteam/templates/workflow_template.yaml .myteam/workflows/planning/launch.yaml
   ```

   You can rename or trim the example steps inside the template to describe the
   workflow you want to run.

3. Start the named workflow with `myteam workflows start planning/launch`. The
   command will now resolve the YAML from `.myteam/workflows/planning/launch.yaml`
   and run the steps the same way the previous path-based entrypoint did.

4. (Optional) If you keep a sample workflow in version control, replace any
   previous ad-hoc `workflow.yaml` files with the new `.myteam/workflows/...`
   layout so future invocations can simply pass the workflow name.

These steps keep the rest of your `.myteam` tree intact while letting agents
discover, scaffold, and run workflows using the same naming pattern as roles and
skills.
